## 2026-03-01 - Fast parsing using iterative line checking instead of Regex
**Learning:** Compiled regex `.finditer()` for parsing large multiline textual blocks (like subtitle SRT files) can be significantly slower than a direct programmatic iterative string traversal. Profiling revealed that an iterative linear traversal with basic `line.isdigit()` and `-->` checks was 2x faster than searching the entire file directly with `block_pattern_obj.finditer()`, and is safer than rigid string `split('\n\n')` approaches which fail if newline spacing between blocks is inconsistent.
**Action:** When extracting predictable blocks from large text documents, prefer using native programmatic iterative parsing over large complex multi-line regex matchers.

## 2026-03-02 - String Parsing and Regex Overhead in Fast Paths
**Learning:** Instantiating `pathlib.Path` objects just to extract file names (e.g. `Path(filename).name`) introduces measurable object instantiation overhead compared to the simpler `os.path.basename`. Additionally, recompiling regular expressions via `re.sub`/`re.match` within frequently called functions adds cache lookup overhead. Benchmarking revealed that switching to `os.path.basename` and using globally pre-compiled regex variables in a hot string-processing loop like `extract_video_id` improved the function speed by over 3.5x (~0.51s down to ~0.14s for 10,000 iterations).
**Action:** In frequently called Python functions, pre-compile regular expressions globally and prefer `os.path.basename` over `Path().name` for simple string parsing to minimize instantiation and function call overhead.
